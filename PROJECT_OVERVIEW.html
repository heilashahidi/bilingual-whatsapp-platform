Nclusion · Field Agent Support

A WhatsApp bridge between 1,000+ field agents and a US support team — with translation built into the middle.

Agents in Haiti, the Dominican Republic, and the DR Congo report issues in their own language on WhatsApp. The US team reads, triages, and replies in English on a web dashboard. Translation, sorting, and routing happen invisibly in between.

API · nclusion-api.up.railway.app Dashboard · nclusion-inbox-production.up.railway.app

01The problem

Why this product needed to exist in the first place.

Nclusion runs branch operations through 1,000+ field agents spread across three countries. When something breaks at a branch — a crashed app, a failed transaction, a connectivity issue — there's no real channel for the agent to flag it. So they don't, or they message someone's personal WhatsApp, or it surfaces days later through a side chat.

Meanwhile the US engineering and operations teams are flying blind. They can't see how often issues happen, which branches are affected, or whether 50 agents are all hitting the same bug at the same time. Even when something does get escalated, the Creole-to-English language gap stretches every back-and-forth from minutes into hours.

02The big idea

Meet agents where they already are. Give the US team a real workspace. Make the language gap disappear.

1
WhatsApp is the front door. Agents don't install anything new — they just message the same WhatsApp number they'd message a colleague. WhatsApp is also the only thing that works reliably on a Haitian 2G connection.
2
A purpose-built inbox for the US team. Not a Slack channel. Not a spreadsheet. A real shared inbox with tickets, statuses, SLAs, and assignments — built like Front or Intercom, but tuned for this exact workflow.
3
Translation is invisible, in both directions. Creole, French, and Spanish come in → English appears on the dashboard. The US team types English → the agent receives it in their language. Neither side has to think about it.
4
Patterns surface automatically. If three agents in Haiti all report the same kind of problem within 30 minutes, the system notices and creates an "incident" — so the team can respond to the root cause instead of triaging the same ticket three times.
5
Every resolved ticket becomes institutional memory. When a ticket closes, the system drafts a knowledge-base article from how it was solved. The next agent who reports the same issue gets a head-start instead of starting from zero.

03The life of one message

What happens between an agent pressing "send" and a US operator seeing the ticket.

  1. Agent sends a WhatsApp message in Creole, French, or Spanish.
  2. Twilio delivers it to our API as a webhook. The API responds instantly so WhatsApp isn't blocked.
  3. The message is de-duplicated so a Twilio retry can't create a phantom ticket.
  4. We look up (or auto-create) the agent by phone number — country comes from the country code.
  5. Claude Haiku translates the message to English and classifies it (category, severity, tags) — in parallel, ~1 second total.
  6. We find or create a ticket — a new message from an open conversation appends; otherwise we open a fresh one.
  7. The knowledge base suggests past resolutions for the same kind of problem.
  8. If three similar tickets have landed in the same country in 30 minutes, we group them into an incident.
  9. Critical and high-severity tickets fire a Slack notification.
  10. Every open dashboard refreshes over Socket.IO — no one has to hit reload.

04What's actually in the product

The features the US team uses every day.

Inbox

Three-pane shared inbox

Sidebar, conversation list, and a ticket drawer that overlays on top — works from the inbox, kanban, and list views identically. @-mentions on internal notes.

Bilingual

One-click language toggle

Default view is English-only. Flip the toggle to see what the agent actually wrote — useful when the translation feels off or when you want to copy a product name verbatim.

Incidents

Auto-detected outages

When three or more tickets in the same country and category hit within 30 minutes, the system opens an incident with a lifecycle (detected → confirmed → mitigating → resolved) and a timeline of contributing tickets.

Knowledge

Self-writing knowledge base

Every resolved ticket gets drafted into a KB article by Claude. The team approves drafts on the /knowledge tab. New tickets see relevant past fixes pinned in the sidebar.

Realtime

Live everything

New ticket arrives → it pops in at the top. SLA timer hits 30 minutes → it turns red. Another teammate resolves the ticket you're looking at → your view updates. No refresh button needed.

SLA

Country-aware SLAs

Haiti and DRC get 1.5× the standard SLA window because the agent's network adds real delay. The clock starts when the agent sent the message, not when our server happened to receive it.

05Where AI shows up

Five places Claude Haiku runs in production — each one fails gracefully if the API is down.

1 of 5

Translation

Inbound messages → English. Outbound English replies → the agent's language. If the message is already English on either side, we skip the LLM entirely and save ~750ms.

2 of 5

Classification

Reads the English text and assigns category (bug, complaint, request…), severity, tags, and product area. Output is structured JSON parsed straight into the database.

3 of 5

Reply drafts

When the operator opens a ticket, three suggested replies appear — varying in tone (direct, empathetic, investigative). They're suggestions, never auto-sent.

4 of 5

Incident summaries

When a cluster forms, Claude rewrites the mechanical title ("Bug report surge — Haiti") into something specific ("Login screen frozen across HT branches") and proposes a root-cause hypothesis.

5 of 5

KB article drafts

On ticket resolution, Claude turns the conversation + the operator's resolution summary into a full KB draft: title, problem description, fix steps. Lands in /knowledge?status=draft for human review.

Privacy

What never leaves our servers

Phone numbers, WhatsApp message IDs, internal user emails, auth tokens, Twilio credentials — none of these ever get sent to Anthropic. Only the message content and a small amount of context.

06Built for the connection agents actually have

Haiti has some of the slowest internet in the Western Hemisphere. The DRC isn't much better. That reality shaped the architecture — it's not an afterthought.

A typical agent might be on 2G with effective speeds under 100 Kbps. Power outages take their phone offline for hours. Messages queue on the device and then deliver all at once when connectivity returns. The system treats this as normal, not as an error case.

What that looks like in the code

Two timestamps on every message

"When the agent pressed send" and "when our server received it." Every clock in the system — SLAs, incident windows, analytics — uses the first one, so a slow network doesn't penalize the team's response time.

Extended SLAs by country

HT and CD get 1.5× the standard windows. DO uses standard timing.

Message-burst tolerance

When ten queued messages deliver in three seconds after an outage, the system uses each message's original send-time to reconstruct the real timeline — so it doesn't look like a fake spike.

Connectivity health monitoring

A background worker watches the delay between "sent" and "received" per region. If a region goes quiet for 30+ minutes, the dashboard surfaces a "possible outage" alert before anyone has to ask.

What it's built with, and the one-line reason for each pick.
